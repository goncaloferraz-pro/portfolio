# Maintenance Management Mobile App — Usability Scoping

A maintenance management SaaS company needed a usability evaluation of their Android mobile app. This engagement involved scoping the study: documenting the app's screen inventory, analyzing the existing wireflow, and producing a usability evaluation proposal. Primary source documents are in formats no longer readable (.docx files in Google Drive); this case is reconstructed from the folder structure and available file descriptions.

## 1. Context

The client was a SaaS company offering a maintenance management platform with a mobile app (Android). The mobile app was the primary interface for field workers performing maintenance tasks — logging issues, completing checklists, updating work orders, and documenting completed work.

**The brief:** Evaluate the mobile app's usability and produce recommendations for improvement. The scope covered the app at version 17/18 (as indicated by the wireflow file naming).

**Engagement type:** UX consulting — scope definition and proposal, with study execution to follow.

## 3. Method

**Screen inventory (decupagem).**
A systematic inventory of all screens in the app at version 17 — cataloging each screen, its function, and its relationship to the user flow. This is the foundational step for usability evaluation design: knowing what screens exist and how they connect before deciding which flows to test.

**Wireflow analysis.**
The existing wireflow (app version 18) was reviewed to understand the intended navigation model and identify where the screen inventory diverged from the designed flow.

**Questionnaire collection.**
A questionnaire was administered to collect baseline user data — likely covering user profile, task frequency, and self-reported pain points — to inform participant recruitment and task scenario design.

**Usability evaluation proposal.**
A formal proposal was produced for the usability study — covering scope, methodology, participant profile, tasks to be evaluated, deliverables, and timeline.

## 6. Reflection

This is the thinnest case in the portfolio — source materials are primarily in DOCX/Google Docs formats that were not exported to portable formats. The engagement is documented here for completeness, but the evidence base is limited to file names and folder structure.

**Key takeaway for documentation practice:** Platform-specific formats (Google Docs, Sketch, XMind, DOCX) are opaque to future review and to portfolio reconstruction. Exporting key deliverables to portable formats (PDF, markdown) at the end of each engagement is a research ops discipline that should be standard — particularly for short or standalone consulting engagements where there is no ongoing client relationship to return to.
